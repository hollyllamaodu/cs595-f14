					<h1>Programming with Member References in NetBeans 8</h1>
					  
					<div class="author colLeft">

<p>Lambdas, introduced in Java 8, have provoked such a storm of discussion, translated into new books, articles, and further research, that their more compact cousins appear to have been left standing somewhat forlornly along the sidelines. This article will discuss these new compact expression styles introduced in Java 8 together with lambdas, collectively known as "member references", which are either "method references" or "constructor references." Migrating from lambdas to member references should be exactly your piece of cake if you like your code even sparser and even more devoid of ceremony than the code that lambdas leave you with in cutting back the boilerplate from your Java code.</p>

<p>When you compare it to human speech, member references are somewhat like dropping the majority of the letters of words and, instead, referring to them by the one or two letters that you know your hearer understands to refer to complete words known to both parties. At first, this mode of speaking is extremely dense and almost like a dialect in itself. However, once familiarized with it, devoid of verbosity, conversation can be extremely quick and to the point.</p>

<p>Ignoring tools where they exist to simplify learning is foolish. Taking NetBeans IDE as a guide to learning about member references is a logical path to take because the same team of developers implementing member references in Java have provided the related code analyzers in NetBeans IDE.</p>
<p>We begin by looking at <em>method</em> references, after which we deal with <em>constructor</em> references.</p>
<h2>Method References</h2>
<p>To begin learning about method references, let's take as a point of departure the <tt>Greeting</tt> class, shown below. It has two different kinds of methods, a static factory method <tt>createGreeting</tt> and an overridden instance method <tt>toString</tt>. Via method references in Java 8, we can learn how to refer to these two methods in new ways to initialize variables with them, to pass them around, and to execute them.</p>
<p><img alt="NB01" height="270" src="/imagesvr_ce/4105/NB01.png" width="448" /></p>
<div class="colLeft dimRelatedArticles_Ads">
	<div class="section relatedArticles">
		<div class="sectionHeader">
			<h3>Related Articles</h3>
		</div>
		<div class="sectionContent">
			<ul>
				<li><a href="/java/new-core-features-in-java-8.html">New Core Features in Java 8</a></li>
				<li><a href="/java/start-using-java-lambda-expressions.html">Start Using Java Lambda Expressions</a></li>
			</ul>
		</div>
	</div>
</div>
<p><strong>Code segment 1</strong></p>
<p>Member references, like lambdas, operate specifically in the context of functional interfaces, the concept of which also is new in Java 8. The familiar <tt>Runnable</tt> class and the <tt>ActionListener</tt> have, retrospectively, been functional interfaces all along because they each provide a single abstract method. Java 8 comes with new functional operations, two of which we will use to illustrate the relevance of member references in this article:</p>
<ul>
<li><strong>java.util.function.Function&lt;P,R&gt;:</strong> The Function interface&nbsp;represents a method that receives a parameter of type "P" and returns a result of type "R".</li>
<li><strong>java.util.function.Supplier&lt;R&gt;:</strong> The Supplier interface represents a method that receives no parameters and returns a result of type "R".</li>
</ul>
<p>Below you see an implementation of the <tt>Function</tt> interface making use of the <tt>Greeting</tt> class shown above. When a parameter of type <tt>String</tt> is received, the <tt>Function</tt> returns a result of type <tt>Greeting</tt> by referring to its method <tt>createGreeting</tt>. In NetBeans IDE, notice the yellow underline and the yellow lightbulb in the left sidebar, informing you that the IDE is aware of a better way to express the <tt>Function</tt> implementation. It is not wrong to use anonymous inner classes in Java 8 and therefore, as you can see below, no red error mark is shown and no compiler error will occur. However, NetBeans IDE, via the yellow lightbulb, is able to offer to help you make the optional conversion to either lambda expressions or, in this case, because reference is made to the method <tt>createGreeting</tt>, to method references.</p>
<p><img alt="NB02" height="273" src="/imagesvr_ce/6495/NB02.png" width="600" /></p>
<p><strong>Code segment 2</strong></p>
<p>When you press Alt-Enter or click the lightbulb, the IDE shows that the anonymous inner class can either be expressed as a lambda expression or as a member reference.</p>
<p><img alt="NB03" height="262" src="/imagesvr_ce/7297/NB03.png" width="600" /></p>
<p><strong>Code segment 3</strong></p>
<p>By invoking the proposal to let the IDE convert the anonymous class to a lambda expression, the following code is automatically created for you.</p>
<p><img alt="NB04" height="166" src="/imagesvr_ce/4733/NB04.png" width="600" /></p>
<p><strong>Code segment 4</strong></p>
<p>Lambda expressions are discussed in our previous article <a href="http://www.developer.com/java/programming-with-lambdas-in-netbeans-8.html">Programming with Lambdas in NetBeans 8</a>, so we will not discuss them again here, because we're only using them here to contrast them with member references. However, note that an alternative, slightly more verbose but possibly more readable, style for stating the lambda expression is offered, too, as shown below.</p>
<p><img alt="NB05" height="167" src="/imagesvr_ce/214/NB05.png" width="600" /></p>
<p><strong>Code segment 5</strong></p>
<p>Here is the result of writing the lambda expression as a block, where a <tt>return</tt> statement declares the right side of the lambda expression more clearly, but less succinctly than in the default lambda expression.</p>
<p><img alt="NB06" height="260" src="/imagesvr_ce/5993/NB06.png" width="600" /></p>
<p><strong>Code segment 6</strong></p>
<p>Having shown the lambda expressions that can be used in the context of our <tt>Greeting</tt> class, let's now contrast them with method references. As you can see below, the IDE can convert the anonymous inner class to a member reference.</p>
<p><img alt="NB03-7" height="262" src="/imagesvr_ce/7867/NB03.png" width="600" /></p>
<p><strong>Code segment 7</strong></p>
<p>Take careful note of the symbol that is used to express a member reference. Instead of the "-&gt;" symbol, a "::" is used instead, as shown below.</p>
<p><img alt="NB07" height="210" src="/imagesvr_ce/1439/NB07.png" width="600" /></p>
<p><strong>Code segment 8</strong></p>
<p>While still referencing the <tt>createGreeting</tt> method, as done by the lambda expressions earlier, the code is considerably more compact, though, until you are used to the style, considerably more dense. To unpack it, look at it carefully and notice it is in the form <tt>Object::Method</tt>. The object here, <tt>Greeting</tt>, contains the method <tt>createGreeting</tt>, which we want to reference. Also notice that, to make the code even more compact, the parentheses of the <tt>createGreeting</tt> method are omitted.</p>
<p>Not only can the static factory method <tt>createGreeting</tt> be referenced via a method reference, but the instance method <tt>toString</tt>, or any other instance method, can be expressed via method references too, as you can see below.</p>
<p><img alt="NB08" height="229" src="/imagesvr_ce/3341/NB08.png" width="600" /></p>
<p><strong>Code segment 9</strong></p>
<p>Above, notice that we are referring to a method that is held by an <tt>instance</tt> of <tt>Greeting</tt>. At the point that we refer to the <tt>toString</tt> method, the <tt>Greeting</tt> instance <em>does not yet exist</em>. Later, in the <tt>apply</tt> method, the <tt>Greeting</tt> instance is created. An alternative style, where you want to refer to methods of <em>existing</em> instances is expressed below, where you see that the <tt>Greeting</tt> is declared and an instance is initialized, after which reference is made to one of its methods. Instead of the <tt>Function</tt> interface, here we use the <tt>Supplier</tt> interface (described above), because we do not want to <em>convert</em> types; instead, we only want to <em>return</em> the result of a method invocation on a <tt>Greeting</tt> instance.</p>
<p><img alt="NB09" height="234" src="/imagesvr_ce/1676/NB09.png" width="598" /></p>
<p><strong>Code segment 10</strong></p>
<p>Note that, where relevant, <tt>super::toString</tt> and <tt>this::toString</tt> are also acceptable constructs.</p>
<h2>Constructor References</h2>
<p>Switching our focus to <em>constructor</em> references, we rewrite the <tt>Greeting</tt> class. Firstly, we remove the static factory method; secondly, we add a default constructor. Now we have two constructors, enabling us to learn how to use constructor references in Java 8 to refer to them and to execute them.</p>
<p><img alt="NB10" height="252" src="/imagesvr_ce/4287/NB10.png" width="306" /></p>
<p><strong>Code segment 11</strong></p>
<p>Below, following similar patterns to those introduced in the section on method references above, are the three new ways introduced in Java 8 for referencing constructors, each of which we will now look at more closely.</p>
<p><img alt="NB11" height="337" src="/imagesvr_ce/6760/NB11.png" width="600" /></p>
<p><strong>Code segment 12</strong></p>
<p>Above, three different constructor reference examples are shown, each discussed below, in turn.</p>
<ol>
<li>The format of a constructor reference is <tt>Object::</tt> new, where <tt>Object</tt> is always the name of the class that defines the constructor. The method name is always the standard Java <tt>new</tt> keyword. Here, we initialize a variable of type <tt>Supplier&lt;Greeting&gt;</tt> with the <tt>Greeting</tt> class default constructor. The compiler knows that we mean the default constructor and not the parameterized constructor because <tt>greetingFactory1</tt>, which is the context of the constructor reference, is of type <tt>Supplier&lt;Greeting&gt;</tt>. The <tt>Supplier&lt;Greeting&gt;</tt> type does not allow parameters and therefore the compiler assumes the reference is the default constructor.</li>
<li>In the second example, the context is <tt>greetingFactory2</tt>, which is of type <tt>Function&lt;String,Greeting&gt;</tt>, which allows parameters of, in this case, type <tt>String</tt>. Therefore, this constructor reference enables us to obtain the reference of the parameterized constructor that accepts a <tt>String</tt> parameter.</li>
<li>The final example is concerned with an <tt>array constructor reference</tt>. As can be seen here, we can create an <tt>Array</tt> an <tt>Greeting</tt> objects. The <tt>Function&lt;int,Greeting[]&gt;</tt> variable type allows a parameter of type <tt>Integer</tt>, and a <tt>Greeting[]</tt> return type. Notice that square brackets are used in the variable type definition, as well as in the constructor reference itself.</li>
</ol>
<p>In this article, we have discussed member references, a new construct introduced in Java 8 together with lambda expressions. Moreover, we have discussed the topic in the context of the code analyzers introduced in NetBeans 8, which help to quickly identify and convert anonymous inner classes to member references. Of these, the most powerful is a batch tool that lets you search through all your code and identify all anonymous inner classes that can be converted to lambda expressions or member references:</p>
<p><img alt="NB12" height="411" src="/imagesvr_ce/9662/NB12.png" width="600" /></p>
<p><strong>Code segment 13</strong></p>
<p>By using the above tool, the IDE analyzes a scope, such as all projects, or the current project, or a package, or file, and suggests code that can be rewritten to use lambda expressions or member references. Not only is this useful for learning about member references, it makes migrating your code to the new compact coding styles made possible in Java 8 a breeze.</p>					<br/>
