							<h2>Why Another MVC?</h2> <p><em>by Ed Burns</em></p> <p><strong>Take a look at  two kinds of MVC and understand their complementary usages.</strong></p> <span style="font-size: smaller;">Published August 2014</span><br /> <br /> <p>The recent filing of a new JSR for an MVC 1.0 framework in Java EE 8   [1] calls for some clarification on how that JSR relates to JSF.  This   document aims to reassure the JSF user community about our continued   commitment to evolving JSF and to clarify the complementary usages of   these two view   technologies. <a href="https://twitter.com/javajoshw">Joshua Wilson</a> <a href="http://javajoshw.blogspot.com/2014/08/java-mvc-10.html">beat me   to the punch</a> with a similar blog entry, which is also well worth   reading.</p> <p>First, let&rsquo;s get one thing straight.  Java EE already has an MVC   framework: JSF.  Why do we need another one?  Isn&rsquo;t this overkill?  The   answer lies in a subtle distinction between different kinds of MVC.  For   the purposes of this article, we consider two kinds of MVC: UI component   oriented MVC and action oriented MVC.  There are other kinds of MV*   concepts out there, but let&rsquo;s stick with just these two.  After looking   at these two different styles of MVC, the article explains the rationale   for filing a separate JSR for MVC 1.0 and explains how this new   specification will relate to the next version of JSF.</p> <p><strong>UI Component Oriented MVC == JSF</strong></p> <div style="float:left"><p><img vspace="10" hspace="10" alt="Diagram
showing how MVC applies to JSF" src="/technetwork/java/jsfcomponent-framework-2280471.png" /><br /> <em>Figure 1: UI Component Oriented MVC as Exposed in JSF</em></p></div> <p>With JSF, the model is CDI, the View is your Facelet pages, and the   controller is the JSF lifecycle, as shown in Figure 1.  One hallmark of   UI component oriented MVC is the emphasis it places on the concept of   &ldquo;inversion of control (IoC)&rdquo;.  JSF takes this concept into the view   tier, allowing you to define little bits of code for your view related   concerns (conversion, validation, events).  This, along with the   existing IoC provided by CDI, means that JSF is much more about setting   up the pieces and letting the framework do the rest (including managing   the view state) than it is about managing the request/response flow of   the underlying HTTP.</p> <br /> <br /> <p>The following stock symbol lookup code example illustrates the   extreme IoC in component oriented MVC by emphasizing the &ldquo;little bits of   code for your view related concerns&rdquo; concept.  First the view:   index.xhtml</p> <pre>
&lt;!DOCTYPE html&gt;
   &lt;html xmlns=&quot;http://www.w3.org/1999/xhtml&quot;
    &nbsp; &nbsp; &nbsp;xmlns:f=&quot;http://xmlns.jcp.org/jsf/core&quot;
    &nbsp; &nbsp; &nbsp;xmlns:jsf=&quot;http://xmlns.jcp.org/jsf&quot;&gt;
   &lt;head jsf:id=&quot;head&quot;&gt;
   &nbsp; &nbsp;&lt;title&gt;Symbol lookup&lt;/title&gt;
   &lt;/head&gt;
   &nbsp;
   &lt;body jsf:id=&quot;body&quot;&gt;
   &nbsp;
   &lt;form jsf:id=&quot;form&quot;&gt;
   &nbsp; &nbsp;
   &nbsp; &nbsp; &lt;p&gt;Enter symbol:
   &nbsp; &nbsp; &nbsp; &nbsp; &lt;input name=&quot;textField&quot; type=&quot;text&quot; jsf:value=&quot;#{bean.symbol}&quot; /&gt;
   &nbsp; &nbsp; &nbsp; &nbsp; &lt;input type=&quot;submit&quot; jsf:id=&quot;submitButton&quot; value=&quot;submit&quot; jsf:actionListener=&quot;#
{bean.actionListener}&quot;&gt;
   &nbsp; &nbsp; &nbsp; &nbsp; &nbsp;&lt;f:ajax execute=&quot;@form&quot; render=&quot;@form&quot; /&gt;
   &nbsp; &nbsp; &nbsp; &nbsp;&lt;/input&gt;
   &nbsp; &nbsp; &lt;/p&gt;
   &nbsp;
   &nbsp; &nbsp; &lt;div jsf:rendered=&quot;#{not empty bean.info}&quot;&gt; 
   &nbsp; &nbsp; &nbsp; &nbsp; &lt;p&gt;Info for #{bean.symbol}: #{bean.info}.&lt;/p&gt;
   &nbsp; &nbsp; &lt;/div&gt;
   &nbsp;
   &lt;/form&gt;
   &nbsp;
   &lt;/body&gt;
   &nbsp;
   &lt;/html&gt;
</pre> <p><br />This is a simple HTML page instrumented with JSF components.  The   symbol to lookup is entered into the text field with name='textField'.  The button to submit the input with type='submit'.  It references an <code>actionListener</code> method.   This is where you put your code that looks up the info for symbol.  This   could interface with a database, a web service, or whatever you like.   Note the <code>&lt;f:ajax /&gt;</code> component inside the form&rsquo;s   button.  This causes the button to be ajaxified, allowing the form   submission and page update to automatically happen over XmlHttpRequest.   Finally, there is a conditionally rendered div that displays the result   of the lookup.</p> <p>The other portion of code is in the Java class referenced by   the <code>#{bean}</code> EL expressions.  This is class Bean.java.</p> <pre>
   import java.io.Serializable;
   import javax.enterprise.context.RequestScoped;
   import javax.inject.Named;
   &nbsp;
   @Named
   @RequestScoped
   public class Bean implements <a href="http://www.google.com/search?hl=en&amp;q=allinurl%3Aserializable+java.sun.com&amp;btnI=I%27m%20Feeling%20Lucky">Serializable</a> 
   &nbsp; &nbsp;
   &nbsp; &nbsp; private <a href="http://www.google.com/search?hl=en&amp;q=allinurl%3Astring+java.sun.com&amp;btnI=I%27m%20Feeling%20Lucky">String</a> symbol;
   &nbsp;
   &nbsp; &nbsp; public <a href="http://www.google.com/search?hl=en&amp;q=allinurl%3Astring+java.sun.com&amp;btnI=I%27m%20Feeling%20Lucky">String</a> getSymbol() {
   &nbsp; &nbsp; &nbsp; &nbsp; return symbol;
   &nbsp; &nbsp; }
   &nbsp;
   &nbsp; &nbsp; public void setSymbol(<a href="http://www.google.com/search?hl=en&amp;q=allinurl%3Astring+java.sun.com&amp;btnI=I%27m%20Feeling%20Lucky">String</a> symbol) {
   &nbsp; &nbsp; &nbsp; &nbsp; this.symbol = symbol;
   &nbsp; &nbsp; }
   &nbsp; &nbsp;
   &nbsp; &nbsp; public void actionListener() {
   &nbsp; &nbsp; &nbsp; &nbsp; // Poll stock ticker service and get the data for the requested symbol.
   &nbsp; &nbsp; &nbsp; &nbsp; this.info= &quot;&quot; + System.currentTimeMillis();
   &nbsp; &nbsp; }
   &nbsp; &nbsp;
   &nbsp; &nbsp; private <a href="http://www.google.com/search?hl=en&amp;q=allinurl%3Astring+java.sun.com&amp;btnI=I%27m%20Feeling%20Lucky">String</a> info;
   &nbsp; &nbsp;
   &nbsp; &nbsp; public <a href="http://www.google.com/search?hl=en&amp;q=allinurl%3Astring+java.sun.com&amp;btnI=I%27m%20Feeling%20Lucky">String</a> getInfo() {
   &nbsp; &nbsp; &nbsp; &nbsp; return info;
   &nbsp; &nbsp; }
   }
   &nbsp;
</pre> <p>This really is just a simple Java object with a read/write JavaBeans   property called <code>symbol</code>, a read only JavaBeans property   called <code>info</code> and a method   called <code>actionListener()</code>.JSF handles all the boilerplate   mechanics of the controller.  The app will deploy easily to any Java EE   7 container, such as <a href="http://glassfish.org/">GlassFish 4.0</a>.   The default URL after deployment is   &lt;<code>http://localhost:8080/symbol/faces/index.xhtml</code>&gt; This   simple example is hosted at   &lt;<a href="https://bitbucket.org/edburns/jsf-symbol">https://bitbucket.org/edburns/jsf-symbol</a>&gt;.   The example includes two additional files not shown here, one of them   the zero-byte marker file: <code>faces-config.xml</code>, the other the   Maven pom.xml.</p> <p>UI component oriented MVC is useful when you are ok with hiding the   HTML/CSS/JS and HTTP from the page author.  The page author is more   about composing UIs from pre-built components and putting in little bits   of code here and there to integrate it with the model tier.  With this   approach, you are indeed placing a lot of trust in the container, but   you get a very simple and rapid development model in return.</p> <p>The benefits of this approach start to diminish when requirements   dictate the need to access the lower level HTTP request/response cycle   and when more control is desired over the rendering and behavior of the   HTML/CSS/JS comprising the UI.  The HTML5 Friendly Markup feature of JSF   2.2 mitigates this particular concern somewhat, but there is no getting   around the fact that component frameworks look at the world through   component colored glasses.  These problems have lead to some to label   JSF as &ldquo;too sticky&rdquo;, but it must be said that this stickiness is a side   effect of IoC and dependency injection in general.  This stickiness also   gets in the way when trying to combine JSF with other technologies   without regard for how JSF operates.  Finally, for one who is already   used to the request/response style of programming, such as those coming   from a PHP or Struts background, the learning curve for JSF requires   changing the way one thinks about web app development.  Such a mindshift   is often perceived as not worth the effort. (Your author humbly   disagrees.)</p> <p><strong>Action Oriented MVC == MVC 1.0</strong></p> <p>In contrast to UI component oriented MVC, the kind of MVC being   advocated in MVC 1.0 can be called &ldquo;action oriented MVC&rdquo;.   This style is shown in Figure 2 and was popularized in frameworks such   as Apache Struts and Spring MVC.</p> <div style="float:right"><p><img alt="Diagram illustrating Action Oriented MVC" src="/technetwork/java/jsfactionframework-2280469.png" /><br /> <em>Figure 2: Action Oriented MVC</em></p></div> <p>The controller dispatches to a specific action, based on information   in the request.  Each action does a specific thing to transform the   request and take action on it, possibly updating the model tier.  This   approach does not try to hide the request/response model of the   underlying HTTP, and it also says absolutely nothing about the specifics   of the HTML/CSS/JS comprising the UI.</p> <p>Though the API for MVC 1.0 is wide open, one piece of prior art that   will certainly influence its design is   the <a href="https://jersey.java.net/documentation/latest/mvc.html">Jersey   MVC</a> system.  The Jersey source code includes numerous examples,   including the Bookstore Webap, which is excerpted here.  Again, we start   with the view, this time a simple JSP page.</p> <pre>

</pre> <p>In this approach, the style reference must be manually inserted with the @import statement in the &lt;head&gt; section.  If the page used JavaScript, a reference to it must also be   inserted.  With JSF, these resources would automatically be included by   virtue of using a component that required them in the page.  In   the <code>symbol</code> example above, the usage of <code>&lt;f:ajax   /&gt;</code> causes a JavaScript reference to be included in the page.   The Bookstore Webapp uses several EL expressions, first in the &lt;h1&gt; element.  This bean is   placed in scope by the Jersey MVC framework.  JSTL iterates over the   collection of books exposed by <code>${it}</code>, in the &lt;c:forEach&gt; element.</p> <p>The <code>index.jsp</code> is rendered implicitly when a GET request   to the root URL of the deployed web app is rendered.  This happens due   to the JAX-RS <code>@Path(&quot;/&quot;)</code> annotation on the following   code</p> <pre>
    @Path(&quot;/&quot;)

The href of those anchors refers to a RESTful URL that is   backed by the <code>getItem( )</code> method in the Bookstore.java   class.  This illustrates the characteristic of action oriented MVC of   having to manually deal with the &ldquo;what happens when the user clicks   something in the UI&rdquo; machinery.  With component oriented MVC, all of   that is handled by the framework.</p> <p>The Bookstore Webapp runs on GlassFish 4.0 and the source code is   available in the Jersey repo on GitHub:   &lt;<a href="https://github.com/jersey/jersey.git">https://github.com/jersey/jersey.git</a>&gt;.</p> <p>This approach is preferred to a component oriented MVC approach when   a greater degree of control over the request/response is required.  Web   designers may favor this approach because they view the abstraction   provided by a component oriented MVC as just getting in the way when   what they really want to do is control the HTML/CSS/JS themselves.   Finally, this approach is a great fit for REST because of its tight   association with the semantics of HTTP.</p> <p><strong>Conclusion</strong></p> <p>The decision to include MVC 1.0 in Java EE 8, while continuing to   enhance and support JSF, was taken based on community feedback from many   sources, including the Java EE 8 Developer Survey [2].  Given the clear   community preference to do MVC, the question became: in what part of   Java EE should the work be done?  Several options were considered before   deciding on filing a separate JSR.  The most obvious choice was to do   the work in the JSF JSR.  One approach would be to have an   alternate <code>Lifecycle</code> implementation.  This would minimize   developer confusion from the perspective of overall Java EE.  On the   other hand, this approach would scare away people who wouldn&rsquo;t go   anywhere near JSF due to its perceived complexity and steeper learning   curve, even if the association was only organizational.  Another choice   was to do the work within the JAX-RS JSR.  JAX-RS is a good fit for   action oriented MVC because it is inherently is close to HTTP and   therefore inherently action oriented.  A downside for this option is   that it&rsquo;s impossible to do server side MVC, even the action   oriented variety, without introducing some level of statefulness to the   system.  Pure RESTafarians abhor state, and thus would oppose polluting   JAX-RS with statefulness.  The last option, of having a separate JSR has   the benefit of not scaring people away by association with JSF.  The   confusion of &ldquo;which one should I use&rdquo; is still a   possibility, but hopefully this article and the work itself will clear   that up.</p> <p>Finally, don&rsquo;t pay too much attention to which JSR is hosting the   work.  It&rsquo;s better to look at Java EE as a cohesive full stack solution   that has both styles built in.  These two view technologies will share   several key components:</p> <ul>     <li>Use of CDI as the model tier</li>     <li>Use of Bean Validation as the validation layer</li>     <li>Use of Expression Language as the glue between the view       and the model</li>     <li>Use of Facelets and/or JSP as the required supported       view declaration languages.</li> </ul> <br /> <p>The MVC 1.0 JSR will also explore using Faces Flows and/or Flash   scope and will likely share elements of JAX-RS.</p> <p>On the JSF side, in addition to whatever changes are necessary to   support MVC 1.0, JSF 2.3 is a community focused release.  Topics to be   addressed include ajax method invocation, numerous clean up and   clarifications, and community requested features such as better support   for Twitter Bootstrap, jQuery and Foundation.</p> <p><strong>About the Author</strong></p> <p>Ed Burns is currently the spec lead for JavaServer Faces, a topic on which Ed recently co-authored a book for McGraw Hill.
